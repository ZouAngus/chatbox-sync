# Chatbox 存储调研\n\n日期：2026-03-24\n\n## 结论\n\n### 桌面端主存储\n\nChatbox 桌面端通过 Electron 主进程暴露 IPC：\n\n- `getStoreValue`\n- `setStoreValue`\n- `getAllStoreValues`\n- `setAllStoreValues`\n\n底层存储使用 `electron-store`。\n\n在 `src/main/store-node.ts` 中：\n\n- `configPath = path.resolve(app.getPath('userData'), 'config.json')`\n- `store = new Store(...)`\n\n因此，桌面端主键值存储实际落在用户目录下的：\n\n- `config.json`\n\n### Blob 存储\n\n`getStoreBlob` / `setStoreBlob` 指向：\n\n- `path.resolve(app.getPath('userData'), 'chatbox-blobs', sanitizeFilename(key))`\n\n因此 blob 文件位于：\n\n- `chatbox-blobs/`\n\n### 关键会话键\n\n从 renderer 存储层可见：\n\n- `chat-sessions-list`\n- `session:<id>`\n\n### Session 结构特征\n\nSession 含：\n\n- `id`\n- `name`\n- `messages[]`\n- `threads[]`\n- `settings`\n- `messageForksHash`\n- `compactionPoints`\n\nMessage 含稳定 `id`。\n\n## 对外挂同步器的意义\n\n### 有利点\n\n1. 存储结构简单明确\n2. 可以直接读取与改写 `config.json`\n3. 会话有稳定 id\n4. 列表键与实体键天然分离\n5. 可先同步文本会话，暂不处理 blob\n\n### 风险点\n\n1. Chatbox 运行时和外挂同时写 `config.json`，可能有覆盖风险\n2. electron-store 写入节奏需要观察\n3. 如果同步 blob，要处理 `storageKey` 对应文件\n4. 会话内部字段较多，MVP 应避免做复杂字段级 merge\n\n## MVP 建议\n\n### 同步粒度\n\n先做“整 session 快照同步”：\n\n- 上传 `chat-sessions-list`\n- 上传每个 `session:<id>`\n- 服务端维护 revision / hash\n- 拉取时按 session 回写\n\n### 冲突策略\n\n- 默认 last-write-wins\n- 若同 base revision 发生分叉，服务端保留 conflict copy\n\n### 第一阶段不做\n\n- blob 同步\n- 实时 websocket\n- 复杂 message 级 patch\n- 双向细粒度 merge\n
